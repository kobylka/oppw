# Executable cross-component contracts

These fixtures are consumed by `tools/validate_contracts.py`. They are not sample UI data and must not be independently reimplemented in tests.

The validator runs this boundary with real components:

```text
fixture → coordination.php / service-control.php → ingest.php → disposable MySQL
        → status.php / analytics.php / mobile-receipt.php / service-control.php
        → Android JsonParser JVM test
```

`fixtures/open-position.json` describes one open position and a lifecycle with deliberately distinct timestamps. `expectations.json` records stable semantic expectations. Dynamic timestamps, specification hashes, decision IDs and authentication tokens are generated by the validator.

When a contract changes, update the producer, PHP, schema/migration, API, Android model/parser, fixture, expectations and assertions in one change set.
